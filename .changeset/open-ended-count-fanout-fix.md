Repeat-count fields are blank by default: an untouched count on a recurring save now emits `recurrenceCount: undefined` (open-ended series) from both the drag-create popover and ActivityFormDialog. NumberInput gains an opt-in `allowEmpty` prop (`value`/`onChange` widen to `number | undefined` on that arm only). `WeekCalendarView` gains an opt-in `recurrenceEditMode` prop: `'server'` preserves a stored row's empty `recurrenceDays` on chip-popover edits instead of transiently fanning the edited chip out across selected weekdays (for consumers whose rows are server-materialized); default `'local'` keeps the pre-wave behavior of adopting the emitted days unconditionally. The drag-create popover now pre-selects the creation day in the days row (an untouched single-day create still emits `recurrenceDays: undefined` — only an explicit extra day or a repeat frequency produces a series). The chip-edit popover's days row now pre-selects a day-less event's own day (an untouched edit save still round-trips `recurrenceDays: undefined`; events with stored days are unaffected). Both calendar popovers (drag-create and chip edit) now cap to the viewport (`max-h` + scroll, `collisionPadding`) instead of overflowing when taller than the available space. Deselecting a weekday from an event's days row now removes the event's chip from that day — an event with an active days selection renders only on the selected days (its start date remains the series time anchor; deselecting every day reverts it to a plain single-day event). Selecting Daily in either calendar popover's Repeat select now presses all seven days in the days row, and manually toggling a day while Daily is selected switches the frequency to Weekly — Daily always means every day.

BREAKING: `TaskTimeFieldsProps.recurrenceCount` is now `number | undefined` and `onRecurrenceCountChange` receives `number | undefined`.
